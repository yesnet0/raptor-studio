Glossary

Raptor's technical vocabulary, in plain language. Hover a term anywhere in the UI for a one-liner; dig in here for the full shape.

Finding lifecycle

final_status
The verdict raptor assigns to each finding after validation. Values: exploitable, likely_exploitable, confirmed_constrained (primitive exists but hard to chain), confirmed_blocked (no viable path given mitigations), confirmed_unverified (analysis incomplete), ruled_out (failed sanity / ruling check), pending.
verdict
Can the bug be triggered? Axis 1 of raptor's two-axis model: exploitable / likely / difficult / unlikely / unknown.
impact
What happens when you trigger it? Axis 2: code_execution / dos / info_leak / resource_exhaustion / data_corruption / unknown. A null-deref can be exploitable × dos; a format-string can be exploitable × code_execution.

Validation stages

Stage A — Pattern triage
Is this a real vulnerability or pattern-matching noise from the scanner?
Stage B — Attacker reach
What does an attacker need to reach the code, and what's in the way?
Stage C — Sanity
Does the code path actually exist and is it reachable from outside? Catches LLM hallucinations.
Stage D — Ruling
Final call: is this test code, does it have unrealistic preconditions, is the model hedging?
Stage E — Binary feasibility
Given the binary's protections (PIE, NX, canary, RELRO, glibc mitigations), is a working exploit actually achievable? Only runs for memory-corruption vulnerabilities.
Stage F — Self-review
Mechanical + critical review of the analysis output. "What did I get wrong?"

Binary exploit anatomy

chain_break
A concrete reason a specific exploitation technique is blocked. Tagged [source] (from code-level analysis) or [binary] (from Stage E feasibility) to distinguish origin.
what_would_help
Per-finding suggestions for turning a blocked exploit viable — e.g. "find a canary leak primitive", "bypass length check via integer wraparound".
exploitation_paths
Ordered {technique, target} pairs describing how to turn a vuln primitive into a working exploit. Example: {"technique": "ROP chain to system", "target": "libc offset 0x50dc0"}.
Protections
Binary hardening: PIE (position-independent executable, defeats absolute-address ROP), NX (no-execute stack), Canary (stack smashing detector), RELRO (read-only GOT).

Scanner output

CWE
Common Weakness Enumeration — MITRE's taxonomy of software weaknesses. E.g. CWE-78 = OS Command Injection, CWE-89 = SQL Injection, CWE-121 = Stack-based Buffer Overflow.
SARIF
Static Analysis Results Interchange Format — SARIF 2.1.0 is the JSON format Semgrep and CodeQL emit. Raptor-studio falls back to parsing SARIF directly when a run has no post-processed findings.json.
dataflow
A path from a source (attacker-controlled input) to a sink (dangerous operation), through zero or more sanitizers and transformations. Raptor validates path satisfiability with Z3 before spending LLM tokens on hypothetical exploits.
Source / Sink / Flow
The three-part proof raptor attaches to each finding. Source = where tainted data enters; sink = where it's used dangerously; flow = the sequence of transformations in between.

Pipelines & lanes

Source analysis lane
Understand → Scan → Validate. Against a code repository. Uses Semgrep + CodeQL + LLM-assisted validation.
Binary fuzzing lane
Fuzz → Crash analysis. Against a compiled binary. Uses AFL++ with autonomous corpus generation, then rr + GDB + LLM reasoning on each crash.
Forensics lane
OSS forensics. Against a public GitHub repository. Uses the GitHub API, GH Archive (via BigQuery), Wayback Machine, and local git to produce an evidence-backed investigation.
Agentic
The all-in-one source-lane workflow: scan → validate → exploit → patch in a single invocation. Highest coverage, highest cost.
